# Product Requirements Document: Undergraduate Business Student Hobbies Survey

## Overview

Build a short web-based survey targeted at undergraduate business students that collects information about their background and leisure hobbies. The app has two pages: a survey form and a results page showing aggregated responses. Responses are stored in a Supabase PostgreSQL database.

---

## Tech Stack

- **Framework:** Vite + React
- **Database:** Supabase (PostgreSQL)
- **Hosting:** Azure Static Web Apps
- **Charting:** Recharts

## Form Behavior

- All fields are required; display inline validation error messages if the user submits without completing a field
- The "Other" hobby text box appears dynamically only when "Other" is checked
- On successful submission, display a thank-you confirmation screen showing a summary of the user's answers
- The thank-you screen includes two buttons:
  - "View Results" — navigates to the results page
- The submit button shows "Submitting..." while the request is in flight
- If the database insert fails, display an error message and keep the form intact

## Results Page

A separate `/results` route displays aggregated, anonymized data from all submissions. Individual responses are never shown — only group-level summaries.

### Charts / Visualizations

1. **Total Responses** — a large number prominently displayed at the top
2. **Year in College** — vertical bar chart, one bar per year option, ordered chronologically
3. **Most Popular Hobbies** — horizontal bar chart, sorted by count descending. For respondents who selected "Other", use the text they entered in `other_hobby` as the hobby label rather than displaying "Other". Normalize casing before counting (e.g. "Pottery" and "pottery" are treated as the same entry). These user-entered hobbies appear as individual bars alongside the standard hobby options.
4. **Top States Represented** — horizontal bar chart showing up to the top 10 states by response count, with percentage labels

Use a charting library (e.g. Recharts) for all visualizations.

### Navigation

- The results page has a "Home" button linking back to the home page
- The survey form has a "View Results" button in the header linking to the results page

## Design & Quality Standards

### Appearance

- Simple, minimalist design throughout
- White or light neutral background with dark body text
- '#DA3BDB' is the primary accent color — use it for buttons, links, active/focus states, borders, and chart colors
- Do not use `#DA3BDB` for body text; it does not meet WCAG AA contrast requirements at small sizes. Use a dark neutral (e.g. `#1a1a1a`) for body text
- For any text rendered in the accent color (e.g. headings or labels), verify contrast passes WCAG AA before using it

### Responsive Layout

The app must be usable on mobile, tablet, and desktop screen sizes. Use a single-column layout on small screens. Do not assume desktop-only use — the survey will likely be completed on phones.

### HTML5 Validation

All pages must validate as HTML5 with no errors using the [W3C Markup Validation Service](https://validator.w3.org/).

### Accessibility — WCAG 2.1 Level AA

All pages must pass WCAG 2.1 Level AA. Key requirements:

- All form inputs must have a programmatically associated `<label>` element
- Inline validation error messages must be programmatically associated with their fields using `aria-describedby`
- The "Other" hobby text input must be keyboard accessible when it appears dynamically — focus should move to it automatically when "Other" is checked
- All images and icons must have appropriate `alt` text
- Color alone must not be used to convey meaning (e.g. error states must include text, not just a red border)
- All interactive elements must be reachable and operable via keyboard
- Use sufficient color contrast for all text — minimum 4.5:1 for normal text, 3:1 for large text

## Deployment

This app is deployed as an **Azure Static Web App** using Vite + React.

Because the app uses client-side routing (`/` and `/results`), a `staticwebapp.config.json` file must be included in the `public/` directory so Azure serves `index.html` for all routes instead of returning 404s on direct navigation or page refresh.

```json
{
  "navigationFallback": {
    "rewrite": "/index.html",
    "exclude": ["/assets/*"]
  }
}
```

This file is automatically copied to `dist/` during the Vite build.
